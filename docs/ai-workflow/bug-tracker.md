# FilmTrip 缺陷问题清单

> **用途**：集中记录用户反馈的所有缺陷，跟踪直至闭环。  
> **维护责任**：@planner 统一登记，@backend/@frontend 等负责处理，@tester 与 @reviewer 验证关闭。

## 记录规范

- **编号规则**：`BUG-YYYYMMDD-序号`，由当日登记顺序生成。
- **状态枚举**：
  - `新建`：用户提出但尚未分析。
  - `已确认`：完成复现与影响分析。
  - `处理中`：已有明确责任人执行修复。
  - `待验证`：修复完成待测试/审查确认。
  - `已关闭`：测试通过并经 @reviewer 与用户确认。
  - `已搁置`：因依赖或策略暂不修复，需备注原因。
- **闭环要求**：必须有 @tester 的验证记录与 @reviewer 的审查确认，用户确认后方可设置为 `已关闭`。

## 处理流程

1. **登记**（@planner）：收到用户反馈后 10 分钟内在本表追加记录，若信息不全需在备注中说明并追踪补充。
2. **分析**（@architect + @backend/@frontend）：确认问题范围、优先级和责任人，更新状态为 `已确认` 并填写责任人。
3. **执行**（相关开发角色）：按标准开发流程修复，状态改为 `处理中`，同步到任务清单（如 `docs/work-plans/2025-11-07/summary.md`）。
4. **验证**（@tester）：完成测试并记录结果，状态改为 `待验证` → 若通过则标记为 `待用户确认`（备注中说明）。
5. **审查与关闭**（@reviewer + 用户）：审查修改并获得用户确认后，将状态改为 `已关闭`，在备注中写明确认时间与方式。
6. **归档**（@documenter）：每周汇总已关闭缺陷至项目归档，保留关键结论与教训。

> **提醒**：任何未闭环的缺陷都必须在每日总结或追踪记录中提及，直至状态更新为 `已关闭` 或 `已搁置`。
